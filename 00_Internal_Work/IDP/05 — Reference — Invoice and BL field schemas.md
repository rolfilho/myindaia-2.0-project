## Commercial Invoice (CI)

The Commercial Invoice is the primary document used for customs declaration, detailing the transaction between the exporter and the importer. It dictates the duties and taxes applied to the shipment.

### Format

Unlike other trade documents, the CI has no universal standard format. It is generated by the exporter's ERP or accounting system, leading to extreme layout variability. It is typically a multi-page document consisting of a header (entities and terms), a body (tabular line items), and a footer (totals and signatures).

### Key Fields for Extraction Schema


| **Field Name**             | **Expected Data Type** | **Description**                                                                            |  |  |  |
| -------------------------- | ---------------------- | ------------------------------------------------------------------------------------------ | - | - | - |
| `invoice_number`           | String                 | Unique alphanumeric identifier assigned by the exporter.                                   |  |  |  |
| `invoice_date`             | Date                   | Issue date. Format varies highly by origin country (MM/DD/YYYY vs DD/MM/YYYY).             |  |  |  |
| `shipper_details`          | Object/Text            | Name, address, and tax ID of the seller/exporter.                                          |  |  |  |
| `consignee_details`        | Object/Text            | Name, address, and tax ID of the buyer/importer.                                           |  |  |  |
| `incoterm`                 | String                 | Three-letter rule specifying freight responsibility (e.g., FOB, CIF, EXW) and named place. |  |  |  |
| `currency`                 | String                 | Currency code (e.g., USD, EUR, BRL) or symbol.                                             |  |  |  |
| `line_items`               | Array of Objects       | The core table containing the individual products being shipped.                           |  |  |  |
| `line_items[].sku`         | String                 | Part number or stock keeping unit.                                                         |  |  |  |
| `line_items[].description` | String                 | Detailed product description.                                                              |  |  |  |
| `line_items[].hs_code`     | String                 | Harmonized System code (6 to 10 digits). Critical for customs.                             |  |  |  |
| `line_items[].quantity`    | Numeric                | Number of units.                                                                           |  |  |  |
| `line_items[].unit_price`  | Numeric                | Price per unit.                                                                            |  |  |  |
| `line_items[].total_price` | Numeric                | `quantity`\*`unit_price`.                                                                  |  |  |  |
| `total_invoice_value`      | Numeric                | The final amount to be paid, including all adjustments.                                    |  |  |  |

### Nuances & OCR Challenges

* **Table Pagination:** The `line_items` table frequently spans multiple pages. The OCR must relate headers to rows across page breaks and ignore running headers/footers.
* **Multi-line Descriptions:** Product descriptions often wrap across multiple lines within a single cell, confusing basic row-extraction logic.
* **Currency Symbol Ambiguity:** Distinguishing between different dollar types (USD, CAD, AUD) requires contextual analysis if the ISO code is not explicitly stated.
* **Language Mixing:** Headers might be in English, while product descriptions and addresses remain in the local language.

## Bill of Lading (B/L)

The Bill of Lading is a legally binding document issued by a carrier to a shipper. It serves three purposes: a receipt of cargo, a contract of carriage, and a document of title.

### Format

The B/L is more standardized than the CI. While ocean carriers (e.g., Maersk, MSC) and freight forwarders have their own templates, they generally follow a standard FIATA-style layout. It is usually a dense, one or two-page document divided into distinct rectangular blocks (boxes) for specific data points.

### Key Fields for Extraction Schema


| **Field Name**                  | **Expected Data Type** | **Description**                                                          |  |  |  |
| ------------------------------- | ---------------------- | ------------------------------------------------------------------------ | - | - | - |
| `bl_number`                     | String                 | Carrier's unique tracking number. Often located in the top right corner. |  |  |  |
| `booking_number`                | String                 | Carrier's booking reference.                                             |  |  |  |
| `shipper_details`               | Object/Text            | Entity exporting the goods.                                              |  |  |  |
| `consignee_details`             | Object/Text            | Entity receiving the goods.                                              |  |  |  |
| `notify_party`                  | Object/Text            | Entity to be notified upon arrival (often a customs broker).             |  |  |  |
| `vessel_voyage`                 | String                 | Name of the ship and voyage number (e.g., "MSC ALINA V. 123N").          |  |  |  |
| `port_of_loading`               | String                 | Origin port (POL).                                                       |  |  |  |
| `port_of_discharge`             | String                 | Destination port (POD).                                                  |  |  |  |
| `containers`                    | Array of Objects       | Details of the physical shipping units.                                  |  |  |  |
| `containers[].container_number` | String                 | 11-character ISO 6346 standard (e.g., HLXU1234567).                      |  |  |  |
| `containers[].seal_number`      | String                 | Security seal identifier.                                                |  |  |  |
| `containers[].package_count`    | Numeric                | Number of pallets, cartons, or bundles inside.                           |  |  |  |
| `containers[].gross_weight`     | Numeric                | Weight of the cargo (requires unit of measure: KGS or LBS).              |  |  |  |
| `containers[].measurement`      | Numeric                | Volume of the cargo, usually in Cubic Meters (CBM).                      |  |  |  |
| `freight_terms`                 | String                 | "Freight Prepaid" or "Freight Collect".                                  |  |  |  |

### Nuances & OCR Challenges

* **Stamps and Signatures:** B/Ls are official titles of ownership. They are frequently stamped (e.g., "ORIGINAL", "COPY", custom authorities) and signed over the printed text, severely degrading OCR accuracy for the underlying data.
* **Watermarks:** Carrier logos are often printed as heavy watermarks in the background, which can be misread as characters by basic OCR engines.
* **Dense Packing:** The "Marks & Numbers" and "Description of Goods" boxes often contain large blocks of unstructured, small-font text squeezed together without clear delimiters.
* **Standardized Identifiers:** Container numbers are prone to single-character misreads (e.g., reading an 'O' as a '0').

## OCR Pipeline Architecture Recommendations

To build a reliable extraction pipeline for these documents, standard text extraction is insufficient.

1. **Preprocessing Layer:** Implement aggressive deskewing and adaptive binarization to handle poorly scanned documents. Use a separate step to detect and remove stamps/signatures before passing the image to the text extractor.
2. **Classification Layer:** Route the document to the correct extraction model. CIs require strong tabular extraction capabilities, whereas B/Ls require strong spatial bounding-box mapping.
3. **Extraction Layer:** Use a combination of layout-aware OCR (to capture bounding boxes) and a multimodal LLM to map the extracted text to your JSON schema.
4. **Post-processing & Validation:** This is the most critical step. OCR output must be programmatically validated against known business rules to flag anomalies.
